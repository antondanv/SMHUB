Техническая база на текущий момент:

- есть единый `docker compose` сценарий для кроссплатформенного запуска `db + backend + frontend`;

# Этап 0. Организация одного репозитория

## Цель этапа

Сделать так, чтобы frontend и backend жили в одном репозитории, но были аккуратно разделены по папкам и сразу были связаны между собой.

## Что сделать в одном репозитории `SMHUB`

Создать базовую структуру:

SMHUB/  
├── backend/  
│   ├── app/  
│   │   ├── main.py  
│   │   ├── core/  
│   │   │   ├── config.py  
│   │   │   └── security.py  
│   │   ├── db/  
│   │   │   ├── database.py  
│   │   │   └── base.py  
│   │   ├── models/  
│   │   ├── schemas/  
│   │   ├── api/  
│   │   └── services/  
│   ├── alembic/  
│   ├── requirements.txt  
│   └── .env.example  
├── frontend/  
│   ├── src/  
│   │   ├── api/  
│   │   │   ├── apiClient.js  
│   │   │   ├── authApi.js  
│   │   │   ├── usersApi.js  
│   │   │   ├── materialsApi.js  
│   │   │   └── homepageApi.js  
│   │   ├── components/  
│   │   ├── layouts/  
│   │   ├── pages/  
│   │   ├── routes/  
│   │   ├── App.jsx  
│   │   └── main.jsx  
│   ├── .env.example  
│   └── package.json  
├── docs/  
│   └── API_CONTRACT.md  
└── README.md

## Распределение

### Разработчик 1

В папке `backend/`:

- создать FastAPI-проект;  
- настроить main.py;  
- добавить CORS;  
- сделать GET /health;  
- подготовить .env.example;  
- подготовить requirements.txt.

### Разработчик 2

В папке `frontend/`:

- создать React + Vite;  
- подключить React Router;  
- создать базовый layout;  
- создать apiClient.js;  
- сделать главную страницу-заглушку;  
- проверить запрос к GET /health.

## Проверка этапа

Frontend должен получить от backend ответ:

{  
  "status": "ok"  
}

После этого можно считать, что монорепозиторий собран правильно и frontend связан с backend.

# Правило веток

На каждую фичу создавайте **одну общую ветку в этом репозитории**.

Например:

feature/auth  
feature/material-catalog  
feature/favorites

В одной ветке можно менять и backend, и frontend, если это одна фича.

---

# Правило готовности фичи

Фича считается готовой только если готово всё:

1. Таблица или модель в backend.  
2. Pydantic-схемы.  
3. API endpoint.  
4. Проверка через Swagger.  
5. React-страница или компонент.  
6. Подключение к реальному API.  
7. Обработка ошибок.  
8. Проверка пользовательского сценария.

Нельзя считать задачу готовой, если есть только backend или только frontend.

# Финальный критерий готовности MVP

MVP готов, если работает такой сценарий:

1. Пользователь регистрируется.  
2. Пользователь входит.  
3. Заполняет профиль: курс, направление, группа.  
4. Загружает учебный материал.  
5. Материал получает статус pending.  
6. Модератор одобряет материал.  
7. Материал появляется в каталоге.  
8. Другой пользователь находит материал через поиск или фильтр.  
9. Открывает страницу материала.  
10. Скачивает файл.  
11. Добавляет материал в избранное.  
12. Ставит лайк.  
13. Ставит оценку.  
14. Оставляет комментарий.  
15. Главная страница показывает последние, популярные и персональные материалы.

Это уже полноценный проект для демонстрации и защиты.
