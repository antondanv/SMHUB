# GitHub Workflow

## Источник правды

- `GitHub Issues` и [GitHub Project "SMHUB Delivery"](https://github.com/users/antondanv/projects/1) — оперативный backlog, статусы и распределение работы.
- `docs/PLAN.md` — roadmap по этапам и фичам.
- `docs/API_CONTRACT.md` — API-контракт.
- `docs/MODELS.md` — модель данных и ограничения.
- `docs/TASKS.md` — архив старого файлового трекинга; больше не используем как основной список задач.

## Уровни планирования

Чтобы не путать стратегию и исполнение, держим 4 уровня:

1. `IDEA.md` — зачем вообще существует проект и в чем его ценность.
2. `PLAN.md` — этапы и крупные фичи roadmap.
3. `GitHub Issues + Project` — конкретные задачи в работе.
4. `Pull Requests` — фактическая реализация задачи или ее части.

Правило простое:

- если обсуждаем направление проекта, открываем `PLAN.md`;
- если хотим понять, кто что делает сейчас, открываем `GitHub Project`;
- если хотим понять, что именно менялось в коде, смотрим `PR`.

## Как работаем вдвоем

1. Перед началом работы берем или создаем `issue`.
2. У issue должен быть один владелец и один проверяющий.
3. Перед началом реализации создаем отдельную ветку под задачу.
4. Каждая активная задача должна лежать в `GitHub Project`.
5. У каждого разработчика одновременно не больше одной большой задачи в `In Progress`.
6. Если задача блокируется, пишем комментарий в issue с причиной и следующим шагом.

## Статусы в Project

- `Todo` — задача готова к работе.
- `In Progress` — задача в реализации.
- `Review` — реализация готова, открыт PR или задача передана второму разработчику на проверку.
- `Done` — задача смержена и проверена.

Дополнительно используем приоритет:

- `Now` — текущие задачи, которые реально берем в работу прямо сейчас;
- `Next` — следующий набор после завершения текущих;
- `Later` — дальние задачи, которые пока не трогаем.

Правило переходов:

- `Todo -> In Progress` — задача взята в работу, ветка создана.
- `In Progress -> Review` — код готов, нужен взгляд второго разработчика.
- `Review -> Done` — PR проверен и изменения влиты.

Если в ревью нашли правки, задача возвращается из `Review` обратно в `In Progress`.

## Формат issue

Для каждой фичи или бага фиксируем:

- цель;
- контекст;
- backend/frontend scope;
- критерии готовности;
- шаги проверки;
- владельца;
- проверяющего.

Рекомендуемые labels:

- `stage:3` ... `stage:7`
- `area:backend`, `area:frontend`, `area:fullstack`
- `enhancement`, `bug`, `documentation`
- приоритет `Now / Next / Later` задается в GitHub Project field, а не лейблом

## Нормальный размер задачи

Задача считается хорошо нарезанной, если:

- ее можно закрыть одним понятным PR или максимум двумя маленькими PR;
- ее можно проверить отдельным сценарием;
- у нее один основной владелец;
- после ее закрытия появляется заметный результат, а не "еще чуть-чуть".

Если задача слишком большая, делим ее:

- по слоям: backend / frontend;
- по сценарию: create / list / detail / edit;
- по интеграции: сначала API, потом UI.

Признаки, что задачу пора делить:

- в issue больше 12-15 реальных пунктов;
- оба разработчика одновременно правят слишком много одних и тех же файлов;
- без второй половины задача вообще не проверяется;
- один PR начинает закрывать пол-этапа.

## Как добавляем новые задачи

Новые задачи появляются из 4 источников:

1. Следующая фича из `PLAN.md`.
2. Баг, найденный во время разработки или проверки.
3. Технический долг, который реально мешает следующей задаче.
4. Организационная задача: деплой, документация, CI, миграция процесса.

Алгоритм добавления новой задачи:

1. Проверяем, есть ли уже похожая issue.
2. Если это часть roadmap, создаем issue и привязываем ее к нужному `stage:*`.
3. Если это баг, создаем `Bug` issue и сразу пишем шаги воспроизведения.
4. Ставим владельца, проверяющего и статус `Todo`.
5. Выставляем приоритет:
   - `Now`, если без этого блокируется текущая работа;
   - `Next`, если это следующий логичный шаг;
   - `Later`, если это уже есть в roadmap, но пока не горит.
6. Если задача блокирует другую, пишем это первой строкой в body или комментарием.

## Как связаны общий план и backlog

`PLAN.md` остается общим планом проекта.

Он отвечает на вопросы:

- какие у нас этапы;
- какие крупные фичи вообще должны появиться;
- в каком порядке мы движемся.

`GitHub Issues` отвечают на вопросы:

- что именно делаем на этой неделе;
- кто владелец;
- как понять, что задача завершена;
- что уже блокирует нас прямо сейчас.

То есть новые этапы и крупные фичи сначала появляются или уточняются в `PLAN.md`, а потом раскладываются на issue в GitHub.

## Что делаем дальше по roadmap

Текущий рабочий порядок сейчас такой:

1. Закрываем `Stage 3A` и `Stage 3B`.
2. После этого берем `Stage 4A` и `Stage 4B`.
3. Затем двигаемся в каталог: `Stage 4C` и `Stage 4D`.
4. Дальше страница материала и управление материалами: `Stage 4E`, `4F`, `4G`.
5. Только после этого переходим к лайкам, комментариям, рейтингу, персонализации и модерации.

## Ветки и PR

- Под каждую задачу создаем отдельную ветку. Не работаем из `main` и не смешиваем несколько задач в одной ветке.
- Одна issue = одна основная рабочая ветка.
- Ветка под задачу: `feature/<issue-number>-short-name`, `fix/<issue-number>-short-name`, `chore/<issue-number>-short-name`
- В PR обязательно указывать `Closes #<issue-number>`
- Один PR должен закрывать одну основную задачу или понятный кусок большой задачи

Перед переводом issue в `In Progress` делаем:

1. Забираем issue.
2. Создаем ветку под нее.
3. Только потом начинаем кодить.

Примеры:

- `feature/3-material-upload-api`
- `feature/6-materials-catalog-page`
- `fix/11-like-counter`
- `chore/1-workflow-cleanup`

Шаблон названия PR:

- `feat: material upload backend (#5)`
- `fix: auth token refresh (#12)`
- `docs: update workflow and project setup (#1)`

## Definition Of Done

Задача считается готовой, когда:

- реализован весь заявленный scope;
- обновлены `API_CONTRACT.md` и `MODELS.md`, если менялся контракт или модель;
- приложены шаги локальной проверки;
- для UI-задач приложены скриншоты или короткое описание сценария;
- чеклист в issue закрыт;
- код прошел ревью вторым разработчиком.

## Ритм работы

В начале сессии:

1. Открыть `GitHub Project` в table view.
2. Отфильтровать `Todo` и `In Progress`.
3. Проверить, не дублирует ли новая работа уже существующую issue.
4. Взять следующую задачу из текущего этапа.

В конце сессии:

1. Обновить статус issue.
2. Оставить короткий комментарий: что сделано, что осталось, есть ли блокеры.
3. Если открыт PR, отметить в issue ссылку на него.
